# Screenshot Guide

Use this guide to capture the most convincing screenshots for GitHub, recruiter outreach, or portfolio submissions.

## Goal

The screenshots should make three things obvious within a few seconds:

- this is a real product, not just a chat demo
- it supports a building-plan review workflow
- it demonstrates full-stack execution and AI product thinking

## Capture setup

- Use the desktop layout, not mobile
- Hide browser bookmarks if possible
- Close DevTools before capturing
- Use one browser window size consistently across all screenshots
- Avoid showing personal email addresses, tokens, or local debug panels
- Keep the app in English

## Recommended order

## 1. Landing / sign-in screen

`What to show`

- the hero headline
- the product framing around plan intake, code knowledge, and coordinated MEP review
- the sign-in area

`Why it matters`

This gives immediate context and shows that the product has a clear use case.

`Caption`

`Landing and authentication flow for the Building Plan Automation Workspace. The product is framed around plan intake, code-reference retrieval, and coordinated MEP review rather than generic chat.`

## 2. Project dashboard

`What to show`

- the active project summary
- usage or readiness cards
- the main workspace shell

`Why it matters`

This tells recruiters the product has structure, state, and operational visibility.

`Caption`

`Project dashboard showing active review context, readiness signals, and the broader operating surface for plan-review work.`

## 3. Plan and reference file intake

`What to show`

- uploaded plan or reference files
- the file strip or project document area
- any project or code-reference context

`Why it matters`

This makes the document-aware workflow obvious and shows that the assistant is grounded in project context.

`Caption`

`Document-aware intake flow where plans, code references, and supporting files are attached to the active project review workspace.`

## 4. Review chat in action

`What to show`

- a realistic prompt
- assistant output that looks structured
- quick tasks or workflow context near the chat if possible

`Why it matters`

This is the screenshot that proves the AI is part of a useful workflow, not just a text box.

`Caption`

`AI-assisted review chat turning project context into structured findings, checklists, and next-step recommendations.`

## 5. Eyes - Brain - Hands board

`What to show`

- the workflow board
- readiness cards
- pipeline actions like plan intake, code checklist, or engineer handoff

`Why it matters`

This is a strong differentiator because it shows product thinking around workflow stages.

`Caption`

`Workflow model for the MVP: read plans, match findings to code knowledge, and convert output into execution-ready handoff work.`

## 6. Audit, observability, or request logs

`What to show`

- request logs
- audit trail
- usage analytics

`Why it matters`

This helps recruiters see engineering maturity and operational realism.

`Caption`

`Operational visibility through audit logs, request tracking, and usage signals to support debugging, accountability, and production readiness.`

## 7. Collaboration or project-management view

`What to show`

- team or member area
- invite flow
- project-level organization

`Why it matters`

This shows the app is built for teams, not just a single-user prototype.

`Caption`

`Project collaboration surface with shared review spaces, member management, and support for multi-user workflows.`

## Best final set

If you only want 4 screenshots on GitHub, use this order:

1. Landing / sign-in screen
2. Project dashboard
3. Review chat in action
4. Audit or observability panel

## Interview tip

When showing screenshots or doing a live walkthrough, narrate them in this order:

1. explain the workflow problem
2. show how the product is organized
3. show the AI-assisted review output
4. end with observability or team features

That order usually lands better with recruiters than starting with the tech stack.
